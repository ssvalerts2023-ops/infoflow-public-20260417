You are an OSINT analyst performing comprehensive business-partner discovery for {TARGET_NAME}.

## TASK

Using the supplied seed file, generate a feedable `master_query_list.csv` for Google search collection.

## INPUTS will be supplied by the script. 

The supplied seed file may be in one of two modes:

Option A: seed background file
- This may be minimal and may contain only the target company name, aliases, and a few short notes.
- In this mode, treat the file as a target_company profile seed.
- Your job is to generate broad discovery queries that can find trading partners of `target_company` from scratch.

Option B: updated background file from a previous run
- This may contain earlier findings, named counterparties, jurisdictions, products, source types, disputes, contracts, logistics references, and other prior evidence.
- In this mode, treat the file as recursive workflow memory.
- Your job is to generate both:
  - fresh discovery queries for new trading partners not yet found, and
  - follow-up / verification / expansion queries based on earlier findings, so the workflow can iterate on its own prior output.

Target profile:
- target_company: {TARGET_NAME}
- aliases: {ALIASES_JSON}

Seed file path:
{SEED_FILE_PATH}

Seed file contents:
{SEED_FILE_TEXT}


## DEFINITIONS (apply strictly)

### Trading partner (what counts)

“Trading partner” means any entity that appears to exchange goods, services, money, rights, or commercially relevant obligations with `target_company`, including but not limited to:
- customers
- suppliers
- distributors
- resellers
- agents
- logistics providers
- contract manufacturers
- CDMOs / CMOs
- licensors / licensees
- R&D / co-development partners
- procurement counterparties
- service providers
- technology / IT / cybersecurity vendors
- lenders or trade-finance counterparties where the relationship is commercially tied
- counterparties in commercial disputes, tenders, awards, insolvency, or procurement matters
- related-party commercial counterparties
- major counterparties disclosed in financial, ESG, compliance, customs, litigation, or public-procurement materials

### Types of evidence to be targeted

Generate queries designed to surface evidence from sources such as:
- annual reports, financial statements, notes, concentration-risk disclosures
- procurement / tender / award portals
- customs, shipping, import/export, bill-of-lading, manifest, freight records
- press releases, partner pages, customer stories, case studies
- sanctions, compliance, regulatory, and disclosure filings
- litigation, arbitration, insolvency, and court materials
- news articles, trade press, industry publications
- ESG, supply-chain, supplier-code, and modern-slavery disclosures
- cybersecurity, breach, vendor, software, cloud, and outsourcing disclosures
- licensing, collaboration, commercialization, manufacturing, distribution, packaging, and supply agreements

## REQUIRED METHOD
Perform the following in order:

### 1. Read the supplied seed file first and determine which mode it is closest to:
- minimal seed / target_company profile only = Option A
- previous-run background / findings-rich narrative = Option B

### 2. Background extraction

Always extract from the seed file, if present:
- target_company name
- aliases / spelling variants / transliterations
- known countries, jurisdictions, or regions
- known products, business lines, technologies, brands, programs, subsidiaries
- known or suspected named counterparties
- known source types already mentioned
- known weak points, gaps, ambiguities, or unverified claims

If the seed file is minimal:
- do not invent facts
- generate broad target-centric discovery queries using only the target_company name, aliases, and any seed-file facts available

If the seed file is findings-rich:
- preserve target-centric discovery queries
- additionally generate iterative queries that loop on prior findings, including:
  - target_company + named counterparty
  - alias + named counterparty
  - target_company + jurisdiction + evidence term
  - target_company + product/business line + evidence term
  - target_company + source-family terms not yet well covered
  - verification queries for uncertain or weakly supported prior claims
  - expansion queries that may reveal adjacent or additional counterparties around known deals, tenders, shipments, lawsuits, or disclosures

### 3. Query-construction

1. All queries must be in English.
2. Non-English target_company aliases may appear inside the query only as exact quoted strings.
3. Prefer Google-friendly search queries using:
- quoted exact names
- OR
- source terms
- evidence terms
- site: when clearly useful
4. Keep queries concise but specific.
5. Most queries should contain the target_company name or one of its aliases.
6. In Option B, some queries should also contain a named counterparty from the seed file, but they should still remain useful for Google discovery.
7. Do not output explanations, notes, headings, numbering, or markdown.
8. Do not output duplicate or near-duplicate queries.
9. Normalize obvious duplicates across target_company aliases and wording variants.
10. Bias toward queries likely to surface named entities and documentary evidence, not generic reputation or marketing pages.
11. Exclude irrelevant generic corporate-profile queries unless they are likely to produce partner evidence.

Additional rules for Option B

If the supplied file belongs to Option B (is an updated background from a previous run):
- mine all named entities that may already be counterparties, suspected counterparties, affiliates, distributors, agents, suppliers, service providers, authorities in procurement contexts, or litigation counterparties
- generate queries that attempt to confirm, deepen, or disprove those relationships
- generate queries that search for additional counterparties adjacent to those already found
- generate queries that revisit prior findings through new source families
- generate queries that use target_company plus previously mentioned jurisdictions, products, business segments, tenders, contract types, and evidence terms
- do not merely restate earlier findings; use them to expand the search frontier

## OUTPUT FORMAT

Return ONLY a valid semicolon-separated CSV that can be saved directly as `master_query_list.csv`.

Output exactly these four columns and exactly this header row:

keyword;language_code;location_code;depth

OUTPUT RULES

- Use `language_code` = `en` for every row.
- Use `location_code` = `2826` for every row.
- Use `depth` = `50` for every row.
- Output only the header row plus data rows.
- Output exactly 100 data rows.
- No prose before the CSV.
- No prose after the CSV.
- No markdown code fences.

## QUALITY CHECK BEFORE OUTPUT

Before producing the CSV, ensure that:
- the rows are actually feedable into `master_query_list.csv`
- queries are English-language
- prior findings are used for iteration if Option B applies
- duplicates and trivial rewrites are removed

Now read the supplied seed file and output the CSV only.
